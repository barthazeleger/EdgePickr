# EdgePickr Deep Review — Codex

Datum: 2026-04-17
Reviewer: Codex
Scope: volledige actuele codebase (`server.js`, `lib/*`, client HTML/JS, scripts, migraties, auth/db/config-paden)
Methode: onafhankelijke review, geen peek naar Claude-document

## Findings

### [P0 Security] Revoked users and demoted admins keep working 30 days because auth trusts JWT claims only
**Files:** `server.js:387-397`, `server.js:6321-6322`, `server.js:6344-6345`, `server.js:7351-7369`

`requireAuth()` only verifies the JWT signature and expiry, then trusts the embedded `id/email/role` claim directly. It never re-reads the user record or re-checks `status`/`role` against the database on authenticated requests. That means:
- a blocked user keeps API access until the 30-day token expires
- a demoted admin keeps admin access until the 30-day token expires
- role/status changes in `/api/admin/users/:id` do not actually revoke existing sessions

This is a real auth-bypass/revocation gap, not a UX detail.

**Fix direction**
- Make `requireAuth()` load the current user row by `req.user.id`
- reject `blocked`/`pending`
- overwrite `req.user.role` from the database, not from the JWT
- optionally add `token_version` / `session_revoked_at` if you want explicit forced logout semantics

### [P0 Security] Stored DOM-XSS is still possible, and localStorage bearer tokens make impact account-takeover level
**Files:** `server.js:7993-8005`, `index.html:2162-2167`, `index.html:3373`, `index.html:3395-3397`, `index.html:3579-3583`, `js/auth.js:4-23`

The client still renders untrusted data into `innerHTML` in multiple places. Two concrete issues:
- `escHtml()` only escapes `&`, `<`, `>`; it does **not** escape quotes, so it is unsafe for attribute contexts (`onclick=`, `title=`, `data-*`)
- some render paths skip escaping entirely, e.g. the pick search/full-list card uses raw `${p.label}`, `${p.match}`, `${p.league}` in `innerHTML`

Because `/api/bets` accepts arbitrary string fields for `wedstrijd` and `markt` with only type checks, a malicious logged-in user can store payloads that later render in the UI. Since the app stores the JWT in `localStorage`, a successful XSS can immediately exfiltrate the bearer token and take over the session.

This is especially dangerous anywhere admin can view broader data or where shared scan/pick payloads are rendered.

**Fix direction**
- stop building interactive UI with inline `onclick`/string HTML for untrusted values
- replace attribute interpolation with DOM APIs (`textContent`, `setAttribute`, `addEventListener`)
- expand escaping if any legacy HTML composition remains, including quotes
- move auth tokens out of `localStorage` if possible (httpOnly cookie or at least shorter-lived token + refresh flow)

### [P0 Security] Push subscriptions are global, so one user can receive another user’s bet/result notifications
**Files:** `server.js:332-347`, `server.js:7391-7404`, `server.js:9764-9772`

`push_subscriptions` are stored without `user_id`, loaded globally, and `sendPushToAll()` broadcasts every payload to every saved endpoint. Any authenticated user can subscribe their browser, and they will then receive notifications generated by unrelated activity, including bet-result payloads with match/market/PnL details.

This is a direct cross-user data-leak channel.

**Fix direction**
- store subscriptions per `user_id`
- subscribe/unsubscribe only within the authenticated user’s scope
- replace `sendPushToAll()` with `sendPushToUser(userId, payload)` and call it explicitly for intended recipients
- audit existing global notifications versus operator-only alerts

### [P1 Correctness] User-scoped result checks can update the wrong bet because `userId` is dropped on outcome write
**Files:** `server.js:6191-6204`, `server.js:6246-6251`, `server.js:8025-8028`, `server.js:9471`, `server.js:9764`, `server.js:9783-9787`

Bet identifiers are app-level `bet_id` values assigned per user by “max current id + 1”, not globally unique database IDs. `checkOpenBetResults(userId)` reads only that user’s open bets, but when it finds a result it calls `updateBetOutcome(bet.id, uitkomst)` **without** passing `userId`.

So a non-admin hitting `/api/check-results` can end up updating whichever row matches the same `bet_id` globally, or hit ambiguous behavior if multiple users have the same `bet_id`. Because `bet_id` is user-local by construction, this is a real cross-user corruption bug.

**Fix direction**
- always pass `userId` into `updateBetOutcome()` from `checkOpenBetResults()`
- long term: stop using per-user synthetic `bet_id` as the mutating lookup key and move API writes to the real DB primary key

### [P1 Correctness] 2FA login can strand users because email delivery failures are swallowed
**Files:** `server.js:359-370`, `server.js:6314-6319`

`sendEmail()` ignores delivery failures and does not inspect non-2xx responses. The login route then immediately returns `{ requires2FA: true }` after awaiting that no-op wrapper. If Resend rejects the request or the mail cannot be delivered, the UI still switches into code-entry mode, but the user never receives the code.

This is not just observability loss; it is a real login dead-end.

**Fix direction**
- make `sendEmail()` return success/failure
- treat non-OK responses as errors
- only return `requires2FA: true` when delivery succeeded
- log/notify failures explicitly so 2FA outages are visible

### [P1 Operational] `migrate-to-supabase.js` is a production-wipe footgun
**Files:** `migrate-to-supabase.js:20-31`, `migrate-to-supabase.js:97-103`, `migrate-to-supabase.js:121`, `migrate-to-supabase.js:154`

This script hardcodes the production Supabase URL, loads privileged credentials, and then performs unconditional destructive deletes (`bets`, `users`, `scan_history`) before reinserting. There is no environment confirmation, dry-run, project guardrail, or “are you sure” barrier.

Even if it is intended as a one-time migration helper, leaving it in this shape inside the active repo is a serious operational hazard.

**Fix direction**
- remove it from the active root if it is no longer needed, or archive it outside the runnable path
- require explicit `--yes-i-know-this-wipes-production` style confirmation
- require environment allowlisting instead of hardcoded production target

### [P2 Quality] `server.js` still duplicates auth/db/config logic that also exists in `lib/*`
**Files:** `server.js:256-397`, `server.js:436-525`, `lib/config.js:9-107`, `lib/auth.js:1-20`, `lib/db.js:10-66`

Core primitives still exist in two places:
- Supabase/config bootstrapping
- JWT/auth middleware
- user/defaultSettings/loadUsers/saveUser
- bets/stat helpers

This is a standing drift risk, and the changelog already documents that several fixes had to be landed in both `server.js` and the shared libs. That means the architecture still has two sources of truth for critical logic.

**Fix direction**
- make `server.js` consume the shared modules rather than re-declare them
- keep one canonical implementation for auth/db/config state
- remove or archive dead duplicate paths once the server is fully migrated

### [P3 Performance] Push fan-out is fully sequential and sits on request-time paths
**Files:** `server.js:346-352`, `server.js:9767-9772`

`sendPushToAll()` awaits every subscription one by one. As subscriber count grows, every notification path becomes O(n) network round-trips in series. Right now this runs directly off request/timer-driven flows such as bet-result processing.

At small scale this is tolerable; at larger scale it becomes latency amplification and partial-failure prone.

**Fix direction**
- batch with bounded concurrency
- decouple notification fan-out from request-time paths
- store per-user subscriptions first, which also reduces unnecessary sends

### [P4 Doctrine] Execution/admin/operator signals still mix “global/operator” and “per-user” semantics in awkward ways
**Files:** `server.js:7391-7404`, `server.js:9329-9366`, `server.js:11127-11152`

Several paths still blur operator-global and per-user behavior:
- push subscriptions are global, not operator-scoped
- notifications mix `user_id = null` globals with user rows
- some admin endpoints expose “all” semantics while others implicitly assume the single-operator admin model

This matters because the doctrine now leans hard on execution truth and operator truth being explicit and canonical. The current state is workable but not yet clean.

**Fix direction**
- define which resources are operator-global and which are user-private
- encode that in schema (`user_id`, `scope`, `owner_role`) rather than conventions
- remove remaining “null means global” ambiguity where possible

## Open Questions

1. Is EdgePickr still intended to support more than one real human operator, or is “single operator + incidental user accounts” the actual model now?
   That answer changes how aggressively push/notifications/admin data should be partitioned.

2. Is there an existing real DB primary key for `bets` exposed nowhere in the API, or is `bet_id` still the main mutation identity?
   If the latter, I would prioritize fixing that before more automation grows around bet mutations.

3. Is `migrate-to-supabase.js` still expected to be runnable, or can it be archived outright?

## Recommended First Action List

1. Fix auth revocation: DB-backed `requireAuth()` with live status/role checks
2. Fix push privacy: per-user subscriptions and targeted send path
3. Fix DOM-XSS class: remove inline-handler HTML composition for untrusted strings
4. Fix `checkOpenBetResults()` to pass `userId` into `updateBetOutcome()`
5. Make 2FA mail delivery explicit and fail closed
6. Archive or hard-guard `migrate-to-supabase.js`
7. Start collapsing duplicate auth/db/config logic into shared libs
