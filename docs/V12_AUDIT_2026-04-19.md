# EdgePickr v12.0.0 · Audit & Release Plan

**Datum**: 2026-04-19
**Auteur**: Claude (Opus 4.7) · 4 parallelle deep-dive subagents + consolidatie
**Doel**: v12.0.0 moet weken stabiel draaien zonder tweaks — een release die data kan verzamelen in plaats van steeds gerepareerd te worden.
**Input**: v11.3.32 productie-state + operator-rapportage "scanner stuurt valse picks, geld op"

## Executive summary

Audits hebben **21 concrete issues** aan het licht gebracht, verdeeld over 4 domeinen: voetbal scan-body, per-sport scans, learning-loop/autotune, en pick-selection pipeline. Drie bevindingen zijn direct geldverlies-veroorzakend (**P0**), acht zijn structureel scheef (**P1**), rest is observability-gat of cosmetisch.

Belangrijkste inzicht: het patroon "altijd 2 Over 2.5 Bet365 picks" is **niet één bug** maar een **stapel van 5 elkaar versterkende effecten**:
1. BTTS gebruikt `cm.over.multiplier` (P0) → over-boost lekt naar BTTS zonder BTTS-eigen learning.
2. DNB en Double Chance missen volledig een kelly-multiplier (P1) → systematisch onder-gestaked → valt uit ranking.
3. Autotune formule `0.70 + wr*1.0` geeft 1.20 boost bij 50% winrate terwijl break-even bij 1.90 odds pas bij 52.6% ligt (P1) → boost zonder edge-bewijs.
4. Sample-threshold `n≥8` voor autotune is statistisch noise (±18pp CI) (P1) → multipliers zijn lottery.
5. 60-70% van picks wordt **silent** gedropt in mkP zonder emit/log (P0 observability) → diagnose is letterlijk blind.

**Conclusie**: eerst bovengenoemde 5 fixen, DAN laat je het model data verzamelen. Deze release (v12.0.0) dicht alle P0/P1 bugs + installeert observability. Verwachte effect: breed-samplend model over alle markten, geen feedback-loop-concentratie meer op één markt.

## P0 — Direct geldverlies-bugs (must-fix)

### P0.1 · BTTS stake-multiplier mismatch
**File**: `server.js:6203, 6208` · **Effect**: structurele mis-staking op BTTS

```js
// Huidige code — FOUT:
mkP(..., bttsYesEdge * 0.22 * (cm.over?.multiplier ?? 1), ...)   // BTTS Yes krijgt Over-boost
mkP(..., bttsNoEdge * 0.20 * (cm.under?.multiplier ?? 1), ...)   // BTTS No krijgt Under-boost
```

Als autotune `cm.over.multiplier = 1.18` (Over presteert goed), krijgt BTTS Yes OOK 18% boost — zonder dat BTTS zelf goed presteert. Over 2.5 en BTTS zijn gecorreleerd maar **niet identiek** (3-0 eindscore: Over ✓ BTTS ✗).

Tegelijk wordt elke gewonnen/verloren BTTS bet wel netjes geschreven naar calib key `football_btts_yes`/`football_btts_no` (via `learning-loop.js:51` + `detectMarket`). Die learning wordt **nooit geconsumeerd** — de scan leest `cm.over` ipv `cm.btts_yes`.

**Fix**: voeg `football_btts_yes` en `football_btts_no` aan `DEFAULT_CALIB.markets` in `lib/calibration-store.js:13-18`. In `server.js:6203/6208` lees `cm.btts_yes?.multiplier ?? 1` en `cm.btts_no?.multiplier ?? 1`.

### P0.2 · mkP silent drops (observability)
**File**: `lib/picks.js:60-157` · **Effect**: 60-70% van picks wordt stilzwijgend gedropt

Geen log/stats bij: `odd < 1.10`, `ep < MIN_EP`, `ep <= ip + 0.03`, `k <= 0.015`, `adaptiveMinEdge fail`, `execution-gate.skip=true`. Typische scan: 30 candidates → 8 final, 22 verdwenen onzichtbaar.

Dat is de reden dat diagnose elke ronde opnieuw zó lang duurt: geen rejection-feedback beschikbaar.

**Fix**: mkP retourneert via een `ctx.emit` of `ctx.dropCounter` zodat aggregate-drops per reden te zien zijn in scan-log. Minimaal: scan-logger per drop-reden (`silent_odd_too_low`, `silent_min_ep`, `silent_ip_margin`, `silent_kelly_zero`, `silent_adaptive_edge`, `silent_execution_gate`). Aggregate telling: `📉 mkP drops: odd=2 · ep=5 · kelly=3 · edge=12 · exec=1 (totaal 23 van 31 candidates)`.

### P0.3 · fxMeta missing op 3 baseball markten
**File**: `server.js:4337, 4341, 4386, 4390, 4420, 4424` · **Effect**: post-scan-gate werkt niet voor die markten

Baseball NRFI, F5 ML en F5 O/U `mkP()` calls geven geen `_fixtureMeta` mee. Gevolg: `applyPostScanGate` heeft geen sleutel om line-timeline te lookuppen → execution-gate is stil niet actief voor deze picks.

**Fix**: voeg fxMeta toe: `{ fixtureId: gameId, marketType: 'nrfi'|'f5_ml'|'f5_total', selectionKey, line }`.

## P1 — Structureel scheef (should-fix)

### P1.1 · DNB en Double Chance missen kelly-multiplier
**File**: `server.js:6260, 6265, 6328, 6332, 6336` · **Effect**: systematisch onder-gestaked

DNB Home/Away en DC 1X/12/X2 hebben `* 0.24` respectievelijk `* 0.16` als stake-factor maar geen `(cm.<key>?.multiplier ?? 1)`. Alle andere markten hebben wel zo'n multiplier. Als Over×1.18 en DNB×1.00 in de ranking strijden, verliest DNB structureel via lagere `expectedEur`.

**Fix**:
- DNB Home: `* 0.24 * (cm.home?.multiplier ?? 1)`
- DNB Away: `* 0.24 * (cm.away?.multiplier ?? 1)`
- DC 1X: `* 0.16 * (cm.home?.multiplier ?? 1)`
- DC 12: `* 0.16` (geen 1:1 key — laat op 1.0)
- DC X2: `* 0.16 * (cm.away?.multiplier ?? 1)`

### P1.2 · Autotune multiplier-formule break-even bug
**File**: `lib/learning-loop.js:67` · **Effect**: 50% winrate op 1.90 odd krijgt 1.20 boost terwijl dat -5% ROI is

```js
mk.multiplier = Math.max(0.70, Math.min(1.20, 0.70 + wr * 1.0));
// wr=0.50 → multiplier=1.20 (MAX boost)
// Maar bij 1.90 gemiddelde odds is break-even 1/1.90 = 0.526
// Dus wr=0.50 = NET VERLIES, niet boost-waardig
```

**Fix**: odds-bewust:
```js
const avgOdds = mk.sumOdds / mk.n || 1.90;
const breakeven = 1 / avgOdds;
const excess = wr - breakeven;
mk.multiplier = Math.max(0.70, Math.min(1.30, 1.0 + excess * 3.0));
```

Vereist kleine schema-toevoeging: track `sumOdds` in calib per markt (of benaderen met running-avg).

### P1.3 · Autotune sample-threshold `n ≥ 8` is noise
**File**: `lib/learning-loop.js:62` · **Effect**: multiplier-waarden tot n=30 zijn gok, geen signal

Voor n=8, 1.90 odd, p=0.5: 95% CI winrate ≈ ±18pp. Een multiplier van 1.20 of 0.70 bij n=8 reflecteert pure variance, geen edge.

**Fix**: `mk.n >= 20` minimum voor multiplier beweging. Onder n=20: multiplier blijft 1.00. Conservatiever maar eerlijker.

### P1.4 · Cumulatieve signal-push zonder absolute cap
**File**: `server.js:5715-5716` (1X2 voetbal) · **Effect**: Sandefjord-class 15-25pp afwijkingen mogelijk

Elk signaal heeft individuele cap (form ±5%, ref ±4%, H2H ±3%, lineup ±5%, pred ±2.5%, congestion ±3%, weather ±3%). Sommatie zonder cum-cap: kan 15-25pp pushen.

**Fix**: `const cumAdj = Math.max(-0.12, Math.min(0.12, posAdj + splitAdj + predAdj + lineupPenalty.home));` (= hard ±12pp max totaal op 1X2).

### P1.5 · Pitcher reliability-factor niet toegepast
**File**: `lib/model-math.js:410-418, 423-439` · **Effect**: rookie-pitcher krijgt zelfde gewicht als 20-start veteraan

`pitcherReliabilityFactor()` retourneert 0.7 bij <15 IP maar wordt **nergens vermenigvuldigd** met het pitcher-adjustment.

**Fix**: `const adj = +(clamped * reliabilityFactor.factor).toFixed(4)` in pitcherAdjustment return-pad.

### P1.6 · dataConfidence te soepel (0-signal-picks komen door)
**File**: `lib/picks.js:74-78` · **Effect**: picks zonder inhoudelijke signal-basis komen door met 40% confidence

Pick met 0 signals krijgt `dataConfidence = 0.40`. Dat betekent: 40% van kelly wordt ingezet op een pure markt-devig kopie zonder model-input.

**Fix**: `if (sigCount === 0) return null;` — geen signals = geen pick. Of: `dataConfidence = 0` bij sigCount=0 (komt dan niet in topPicks).

### P1.7 · auditSuspicious-logica onvolledig
**File**: `lib/picks.js:102-104` · **Effect**: sommige fake-edges ontsnappen

`probGap > 15 && Math.abs(baseGap) > 15 && Math.abs(signalContrib) < Math.abs(baseGap) * 0.3`

Situatie: probGap=34, baseGap=19, signalContrib=4 → detected ✓. Maar: probGap=30, baseGap=25, signalContrib=10 → signalContrib>0.3×baseGap → **niet gedetecteerd** ondanks grote gap.

**Fix**: OR-logica:
```js
const probGapSuspect = probGap > 15 && signalContrib < probGap * 0.25;
const baseGapSuspect = Math.abs(baseGap) > 12 && Math.abs(signalContrib) < Math.abs(baseGap) * 0.3;
const auditSuspicious = probGapSuspect || baseGapSuspect;
```

### P1.8 · revertCalibration asymmetrisch (geen modelLog entry)
**File**: `lib/learning-loop.js:163-208` · **Effect**: audit trail toont boost-events zonder revert-events → verwarrend

Niet geldverlies maar wel trust-issue. `updateCalibration` logt multiplier-bewegingen in `c.modelLog`. `revertCalibration` update wel `mk.n/w/profit` terug maar schrijft geen revert-entry in modelLog.

**Fix**: voeg revert-entry toe aan modelLog.

## Succes-criteria voor v12.0.0

Na één week draaien moet gelden:

1. **Spreiding**: picks komen uit minstens 3 verschillende sport-markt combinaties per dag (niet meer 2x Over 2.5 Bet365)
2. **Observability**: elke pick-drop heeft een zichtbare reden in scan-log of near-miss UI
3. **Calibratie-consistentie**: geen markt heeft `cm.<key>.multiplier > 1.10` zonder `n >= 20` en `wr > 1/avgOdds`
4. **Geen fake-edges**: geen BTTS picks waar model-prob > 15pp van marketdevig afligt
5. **Staking-gelijkheid**: DNB en DC picks worden met identieke kelly-multipliers gestaked als O/U/1X2 (geen stille under-stake)
6. **Drop-telemetry**: scan-log toont aggregate `mkP drops: odd=X · ep=Y · kelly=Z · edge=A · exec=B`

Als minstens 5 van 6 criteria na een week waar zijn, is v12.0.0 stabiel.
